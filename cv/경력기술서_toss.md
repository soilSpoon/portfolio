## 1. 출근노트 — 근태 및 급여 관리 SaaS

**기간**: 2020.08 – 2025.03
**소속**: 씨엠유니버스(주) | 풀스택 개발자
**제품**: 중소기업 대상 근태/급여 관리 B2B SaaS. 어드민·노무법인·사업주·중간관리자·근로자 5단계 멀티테넌트 구조. 프론트엔드 전체를 1인 주도 개발.

### 1-1. React 디자인 시스템 및 컴포넌트 아키텍처 구축

#### 이슈 사항

초기 개발 시 디자인 시스템이나 컴포넌트 공통화를 고려하지 않고 기능 위주로 개발이 진행되었습니다. 4년 이상 운영되면서 유사한 UI 패턴이 각 페이지마다 독립적으로 구현되어 있었고, 동일한 기능의 테이블이 여러 파일에 각기 다른 방식으로 존재했습니다.

#### 문제점

- 복사-붙여넣기 방식의 UI 구현으로 페이지별 디자인 불일치 발생
- 하나의 버그 수정이 동일 패턴이 사용된 모든 파일에 수동으로 동일 수정을 요구
- 상태관리 패턴이 통일되지 않아 — useState, Context API, 전역 변수가 혼재

#### 개선 사항

- **Chakra UI** 기반 디자인 시스템을 도입하고, 프로젝트 전용 공통 컴포넌트 라이브러리를 추출 및 표준화
- Chakra UI의 **WAI-ARIA 접근성** 지원을 활용하여 키보드 내비게이션, 스크린리더 호환, 포커스 관리를 공통 컴포넌트에 표준화
- 상태관리 계층을 명확히 분리:
  - **React Query** — 서버 상태 (API 캐싱, 갱신, 무효화)
  - **Zustand** — 전역 UI 상태 (사이드바, 모달, 알림)
  - **Jotai** — 컴포넌트 로컬 상태 (폼 단계, 필터 조건)
  - **Zod** — 런타임 데이터 유효성 검증

#### 적용 결과

- 공통 컴포넌트 조합만으로 신규 페이지 구성 가능 — **개발 시간 체감 50% 단축**
- 이 아키텍처 위에 노무법인 시스템 6개 모듈을 안정적으로 신규 개발 완료
- 상태관리 계층 분리로 불필요한 리렌더링 제거, 코드 리뷰 시 상태 흐름 파악 용이
- WAI-ARIA 기반 접근성 적용으로 키보드만으로 주요 업무 수행 가능

### 1-2. 대량 급여 정산 화면의 프론트엔드 성능 최적화

#### 이슈 사항

초기 개발 시 소규모 사업장(수십 명)을 기준으로 설계되어, 서비스 성장 후 수천 명 규모의 사업장이 유입되면서 성능 문제가 발생했습니다. 급여 정산 기간에 수천 명의 근로자 데이터를 한 화면에서 조회·계산·수정해야 했으나, 대량 데이터를 고려하지 않은 설계로 테이블 렌더링 시 **UI가 수 초간 프리징**되어 사용자가 "화면이 멈췄다"는 CS를 반복 접수했습니다.

#### 문제점

- 전체 근로자 데이터를 한 번에 DOM에 렌더링하여 DOM 노드 수천 개 생성 → 브라우저 메인 스레드 블로킹
- 급여 계산 중 부모 컴포넌트 상태 변경이 전체 테이블 행의 불필요한 리렌더링을 유발

#### 개선 사항

- react-window 기반 가상화 테이블로 화면에 보이는 행만 렌더링
- React.memo로 행 컴포넌트를 메모이제이션하고, useMemo로 계산 결과 캐싱
- React Query의 staleTime/cacheTime을 조정하여 동일 데이터의 중복 API 호출 제거

#### 적용 결과

- 정산 페이지 **UI 프리징 현상 완전 해소** — 수천 행 데이터에서도 부드러운 스크롤
- 정산 기간 중 관련 **CS 문의 해소**

### 1-3. B2B2C 5단계 권한 체계의 프론트엔드 구현

#### 이슈 사항

출근노트는 B2B2C 구조로, **어드민 → 노무법인 → 사업주 → 중간관리자 → 근로자** 5단계 권한이 존재합니다. 같은 화면이라도 사용자 레벨에 따라 보이는 데이터, 사용 가능한 기능, 네비게이션 구조가 모두 달라야 했습니다.

#### 문제점

- 권한 분기 로직이 컴포넌트 내부 곳곳에 산재 — 새로운 페이지 개발 시 권한 체크를 빠뜨리기 쉬운 구조
- 새로운 권한 레벨 추가 시, 기존 모든 페이지를 검토하여 권한 분기를 추가해야 하는 높은 변경 비용
- 프론트엔드에서만 권한을 제어하면 API를 직접 호출하여 데이터에 접근할 수 있는 보안 취약점 존재

#### 개선 사항

- **권한 제어 컴포넌트**와 **권한 확인 훅**을 구현하여 선언적으로 권한 기반 렌더링 — 조건부 로직을 컴포넌트 외부로 추출
- 라우트 레벨에서 **권한 가드 미들웨어** 적용 — 접근 불가 페이지는 자동 리디렉션
- 이 아키텍처 위에 **노무법인 시스템 전체**를 신규 개발: 대시보드, 부관리자 관리, 기본정보 관리, 계정 찾기, 근로자 정보 관리, 사업장 검색

#### 적용 결과

- 권한 관련 코드의 **중앙 집중화**로, 노무법인 전체 시스템(6개 모듈)을 기존 프로젝트에 안정적으로 추가 완료
- 신규 권한 레벨 추가 시 권한 제어 컴포넌트 설정만 변경하면 되어 **영향 범위 최소화**
- 프론트엔드 권한 제어의 일관성 확보로 권한 관련 버그 방지

### 1-4. React 기반 신규 프로젝트 기술 스택 선정 및 아키텍처 구축

#### 이슈 사항

입사 당시 회사의 기존 프로젝트들은 jQuery 기반이었습니다. 출근노트는 새로 시작하는 프로젝트였기 때문에, 기술 스택을 직접 선정할 기회가 있었습니다.

#### 문제점

- 기존 jQuery 기반 프로젝트들에서 DOM 직접 조작으로 인한 상태 불일치, UI 재사용 불가 등의 한계를 경험
- 신규 프로젝트에서 같은 문제를 반복하지 않기 위해 모던 프론트엔드 스택 도입이 필요

#### 개선 사항

- 신규 프로젝트에 **React 도입을 건의**하고, 처음부터 컴포넌트 기반 UI 개발 체계로 시작
- 이후 프로젝트 성장에 맞춰 Emotion → Chakra UI → Jotai/Zustand/React Query 등 단계적으로 모던 스택 확장

#### 적용 결과

- 컴포넌트 재사용과 상태 관리를 처음부터 적용하여 **4년간 안정적으로 확장 가능한 프론트엔드 기반** 확보
- 이 아키텍처가 디자인 시스템(1-1), 성능 최적화(1-2), 권한 체계(1-3) 등 모든 개선의 기반이 됨

## 2. 에브리드론 — 클라우드 CAE 시뮬레이션 플랫폼

**기간**: 2025.03 – 2026.01
**소속**: 주식회사 에브리심 | 제품개발 책임
**제품**: CAE(Computer-Aided Engineering) 시뮬레이션을 브라우저에서 실행할 수 있는 클라우드 웹 플랫폼. CAD 모델 업로드부터 메시 생성, 시뮬레이션 실행, 결과 확인까지 전 과정을 설치 없이 브라우저에서 수행.
**기술**: Next.js, TypeScript, Nest.js, WebAssembly, Prisma, PostHog, Storybook, Mastra AI SDK

### 2-1. 프론트엔드 아키텍처 재설계 및 제품 완성도 향상

#### 이슈 사항

제품 출시를 앞두고 기존 프론트엔드 아키텍처에 근본적인 구조 문제가 있었습니다. 시뮬레이션 워크플로우의 각 단계(CAD 로딩, 메시 생성, 시뮬레이션 실행)가 WASM 모듈과 강하게 결합되어 있어, 단계 간 순서 의존성이 코드에 하드코딩된 상태였습니다. 또한 UI 일관성 부재, 인증 시스템 미비 등 제품 완성도를 높이기 위한 전방위적 개선이 필요했습니다.

#### 문제점

- 시뮬레이션 워크플로우의 각 단계가 WASM 모듈의 내부 상태에 직접 의존 — 단계 순서를 변경하거나 독립적으로 실행하는 것이 불가능한 구조
- JavaScript↔WASM 간 강한 결합으로 한쪽을 수정하면 다른 쪽도 반드시 수정해야 하는 높은 변경 비용
- UX를 고려하지 않은 UI 설계로 사용자가 워크플로우 진행 상태를 파악하기 어려움
- 컬러 팔레트, 컴포넌트 스타일이 통일되지 않아 제품 완성도 부족

#### 개선 사항

- 시뮬레이션 워크플로우를 **WASM 모듈과 분리된 상태 관리 계층**으로 재설계 — 각 단계가 독립적으로 실행·테스트 가능하도록 의존성 역전
- JavaScript↔WASM 간 **명확한 인터페이스 계층**을 도입하여 결합도 감소 — WASM 내부 변경이 UI 코드에 전파되지 않는 구조
- UX를 고려한 워크플로우 UI 재설계 — 단계별 진행 상태 시각화, 에러 피드백 개선
- **통합 컬러 팔레트 시스템**으로 전체 UI 색상 체계를 재설계하고, shadcn 컴포넌트를 재구성하여 디자인 시스템 표준화
- **Storybook** 통합으로 컴포넌트 단위 개발·문서화·인터랙션 테스트 환경 구축

#### 적용 결과

- WASM-JS 분리로 워크플로우 단계 간 **독립적 개발·테스트 가능** — 기능 추가 시 영향 범위 최소화
- UX 재설계로 사용자가 시뮬레이션 진행 상태를 직관적으로 파악 가능
- 디자인 시스템 표준화로 UI 일관성 확보 및 신규 화면 개발 속도 향상

### 2-2. OpenCASCADE WASM 빌드 및 브라우저 이식

#### 이슈 사항

3D CAD 모델링에 필요한 OpenCASCADE 엔진을 브라우저에서 실행하기 위해 WebAssembly로 컴파일해야 했습니다. 기존 Emscripten 3.x 기반 빌드는 32비트 제한과 TypeScript 지원 부재 등의 한계가 있었습니다.

#### 문제점

- Emscripten 3.x의 32비트 메모리 제한으로 대형 CAD 모델 처리 시 메모리 부족 발생
- JavaScript↔C++ 간 바인딩에 TypeScript 타입 정의(d.ts)가 없어 프론트엔드 개발 시 타입 안전성 부재
- CMake 빌드 체계와 Emscripten 타겟 간 호환성 문제

#### 개선 사항

- **Emscripten 버전을 3 → 4 → 5로 단계적 업그레이드** 수행
- 64비트 메모리 지원 활성화로 대형 CAD 모델 처리 가능
- **TypeScript 타입 정의(d.ts) 자동 생성** 파이프라인 구축으로 프론트엔드 타입 안전성 확보
- React 컴포넌트의 WASM 로딩 패턴을 전환하여 **Next.js SSR 호환성** 확보 — dynamic import + Suspense로 hydration 경고 제거

#### 적용 결과

- 데스크탑 설치 없이 브라우저에서 3D CAD 모델링 가능 — **사용자 진입 장벽 완전 제거**
- 64비트 지원으로 대형 모델 처리 시 메모리 제한 해소
- d.ts 자동 생성으로 프론트엔드-WASM 간 타입 안전한 인터페이스 확보

### 2-3. PostHog 기반 제품 분석 체계 구축 및 데이터 기반 의사결정

#### 이슈 사항

초기 제품 단계에서 사용자가 시뮬레이션 워크플로우의 어느 단계에서 이탈하는지, 어떤 기능을 주로 사용하는지에 대한 데이터가 전혀 없어 제품 방향성 결정이 감에 의존하고 있었습니다.

#### 문제점

- 사용자 행동에 대한 정량적 데이터 부재로 제품 우선순위 결정이 주관적
- 신규 기능 출시 시 영향도를 측정할 수 없어 반복적 개선이 불가능
- 서버 사이드 이벤트(회원가입, 시뮬레이션 제출 등)가 추적되지 않아 전체 사용자 여정을 파악할 수 없음

#### 개선 사항

- **PostHog 분석 파이프라인을 처음부터 구축**: 클라이언트 초기화 → 사용자 식별 → 서버 사이드 이벤트 추적까지 단계적으로 확장
- 기존 Slack 웹훅 알림을 **PostHog 서버 사이드 이벤트**로 전환 — 회원가입, 로그인, 시뮬레이션 제출 등 핵심 이벤트를 통합 대시보드에서 분석
- 클라이언트 IP 추적으로 지역별·언어별 사용 패턴 분석 기능 추가

#### 적용 결과

- 사용자가 어떤 패턴으로 제품을 사용하고 **어느 단계에서 이탈하는지** 정량적으로 파악 가능 — 데이터 기반 제품 방향성 결정 체계 확립
- **어떤 국가·언어 사용자들이 주로 유입되는지** 파악하여 다국어 대응 및 타겟 시장 우선순위 도출

### 2-4. AI 기반 시뮬레이션 워크플로우 챗봇 프론트엔드 개발

#### 이슈 사항

CAE 시뮬레이션은 **유체 영역 생성 → 경계 조건 설정 → 메시 생성 → 시뮬레이션 실행 → 결과 확인**의 다단계 워크플로우로 구성됩니다. 전문 지식이 없는 사용자가 각 단계의 파라미터를 올바르게 설정하기 어려워 사용 진입 장벽이 높았습니다.

#### 문제점

- 시뮬레이션 파라미터 설정에 CAE 도메인 지식이 필요하여 비전문가의 독립 사용이 어려움
- 각 단계 간 상태 의존성이 복잡 — 유체 영역 변경 시 경계 조건과 메시가 모두 무효화되는데, 사용자가 이를 인지하지 못함

#### 개선 사항

- **AI SDK v5** 기반으로 챗봇 UI를 구현하고, 메시 실행·시뮬레이션 실행·메시 품질 조회 등 **클라이언트 사이드 도구 실행** 구조를 설계 — 챗봇 인터페이스에서 실제 시뮬레이션을 직접 제어
- 스트리밍 응답 렌더링, 도구 실행 상태 표시, 대화 히스토리 관리 등 **챗봇 전용 React 컴포넌트** 개발
- YAML 기반 프롬프트 관리 시스템과 의도 분류(intent classification) UI 구현

#### 적용 결과

- 비전문가도 자연어 대화만으로 시뮬레이션 파이프라인을 조작할 수 있어 **사용 진입 장벽 대폭 감소**
- 모듈러 도구 아키텍처로 새로운 시뮬레이션 기능 추가 시 도구만 구현하면 챗봇이 자동 활용

## 3. GSS — 포스코 철판 결함 분석 시스템 (MES 차공정)

**기간**: 2019.12 – 2023.11
**소속**: 씨엠유니버스(주) | 풀스택 개발자 (MES 차공정 주도)
**제품**: 포스코 생산 라인에서 발생하는 철판 결함 데이터를 실시간으로 수집·시각화하고, 코일 단위 품질 등급을 자동 판정하는 Electron 기반 데스크탑 시스템.
**기술**: Electron, TypeScript, React, Next.js, Ant Design, Laravel, MySQL, WebSocket

### 3-1. Electron + React 실시간 대량 데이터 시각화

#### 이슈 사항

포스코 생산 라인에서 WebSocket으로 실시간 유입되는 철판 결함 데이터를 Electron 앱 내 React 대시보드에 즉시 렌더링해야 했습니다. 결함의 2차원 좌표, 크기, 등급 등을 시각화하고, 코일 단위로 품질 등급을 실시간 판정해야 했습니다.

#### 문제점

- WebSocket으로 유입되는 데이터 속도가 클라이언트 측 가공 속도를 초과 — 데이터가 쌓이면서 렌더링이 점점 지연
- 동일한 결함 데이터에 대해 여러 컴포넌트에서 중복 계산이 발생
- 모든 중간 가공 데이터를 디스크에 저장하고 있어, 운영 기간이 길어질수록 디스크 사용량이 지속 증가
- 코일 등급 판정 기준이 공정마다 상이하여 조건 분기가 매우 복잡

#### 개선 사항

- 수신 데이터에서 불필요한 항목을 걸러내는 **전처리 필터링 단계** 추가
- 가공 결과를 **메모이제이션**하여 동일 데이터의 재계산 방지, **배치 렌더링**으로 DOM 업데이트 최소화
- 중간 데이터를 디스크에 저장하지 않고 **메모리 내 파이프라인 방식**으로 전환 — 최종 결과만 영구 저장
- 등급 판정 로직을 **규칙 엔진 형태로 추상화** — 조건과 결과를 분리하고, 공정별 기준을 설정 파일로 관리

#### 적용 결과

- 데이터 유입 속도에 밀리지 않는 **실시간 처리 성능 확보**
- 메모리 파이프라인 전환으로 운영 중 디스크 사용량이 시간에 따라 증가하던 문제 해결
- 규칙 엔진 도입으로 신규 공정 대응 시 **코드 수정 없이 설정 파일만으로 대응** 가능

## 4. 오픈소스 기여

**기간**: 2020 – 현재

### 4-1. 프론트엔드 빌드 도구 및 번들러 내부 기여

#### 이슈 사항

출근노트 프로젝트의 Laravel Mix 빌드 환경에서 React Fast Refresh가 정상 작동하지 않아 코드 변경마다 전체 페이지가 새로고침되었습니다. 프론트엔드 개발 시 상태가 초기화되어 생산성이 크게 저하되는 문제였습니다.

#### 개선 사항

라이브러리 소스 코드를 직접 분석하여 문제의 근본 원인을 파악하고, 수정사항을 제출:

- **laravel-mix**: **webpack 내부 설정 직접 수정** — React Fast Refresh 기능 추가, webpack-dev-server 4.0 마이그레이션, HMR HTTPS 옵션 복구, BabelConfig 최적화 등 12개 PR merged
- **react-refresh-webpack-plugin**: 호환성 개선 — 3개 PR merged

### 4-2. 프론트엔드 핵심 인프라 기여

- **SWC**: CI GitHub Actions Node 버전 업데이트
- **Jotai**, **Valtio**: 브랜치명 master→main 전환 등 프로젝트 표준화 기여
- **Astro**: Vercel 어댑터에 정적 에셋 캐시 헤더 추가 — CDN 캐싱 성능 개선
- **Snowpack**: React Fast Refresh 최소 버전 요구사항 업데이트
- **Tremor**: 차트 컴포넌트에 null 값 구간 연결 옵션 추가
- **twin.macro**: Tailwind transform 유틸리티 지원 추가
- **vite-plugin-checker**: 소켓 URL 해석 로직 수정
- **webpack/less-loader**: Less 4.0 peerDependency 추가
- 기타: laravel/framework, aleph.js, pest, tighten/collect, NextAuth 등

### 4-3. AI/LLM 도구 생태계 기여

- **CLIProxyAPI**: Gemini↔Claude API 호환 프록시 — 에러 핸들링, 모델 호환성, JSON 스키마 개선
- **Rivet sandbox-agent**: 코딩 에이전트 샌드박스 스키마 확장

#### 적용 결과

- laravel-mix 사용자 전체에 React Fast Refresh 기능 제공
- 프론트엔드 빌드 도구와 상태관리 라이브러리의 **내부 동작 원리에 대한 깊은 이해** 확보
