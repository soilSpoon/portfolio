## 1. 에브리드론 — 클라우드 CAE 시뮬레이션 플랫폼

**기간**: 2025.03 – 2026.01
**소속**: 주식회사 에브리심 | 제품개발 책임
**제품**: CAE(Computer-Aided Engineering) 시뮬레이션을 브라우저에서 실행할 수 있는 클라우드 웹 플랫폼. CAD 모델 업로드부터 메시 생성, 시뮬레이션 실행, 결과 확인까지 전 과정을 설치 없이 브라우저에서 수행.

### 1-1. C++ 메시 생성 엔진 레거시 코드 리팩토링

#### 이슈 사항

CAE 시뮬레이션의 핵심인 메시 생성 엔진이 레거시 C++ 코드로 작성되어 있었습니다. 단일 파일에 모든 로직이 집중된 모놀리식 구조여서 유지보수와 기능 확장이 어려운 상태였습니다.

#### 문제점

- 단일 소스 파일에 메시 생성, CAD 로딩, CLI 파싱, 유체 영역 생성 등 모든 로직이 혼재
- 매크로 기반의 상수 정의로 타입 안전성이 없고, 디버깅이 어려움
- STEP/BREP(파라메트릭)과 STL(이산) 형식의 CAD 입력을 처리하는 코드가 분산되어 새로운 형식 추가 시 전체 코드를 수정해야 하는 구조

#### 개선 사항

- 모놀리식 구조를 메시 생성, 파이프라인, CAD 로딩, 유체 영역 생성 등 **역할별 모듈로 분해**
- **Policy-based design** 패턴을 도입하여 파라메트릭(STEP/BREP)과 이산(STL) 형상을 통합 처리하는 구조 설계 — 새로운 CAD 형식은 정책 구현만으로 추가 가능
- C++ 매크로를 타입 안전한 상수로 전환하고, 직접 벡터 접근으로 성능 최적화
- 유체 영역 생성(Box/Cylinder) 워크플로우를 통합하고, **clang-tidy 기반 C++ 정적 분석** 파이프라인 구축

#### 적용 결과

- 모듈 단위 독립 빌드·테스트가 가능해져 **기능 추가 시 영향 범위 최소화**
- Policy-based design으로 신규 CAD 형식 지원 시 기존 코드 변경 없이 정책만 추가
- 유체 영역 생성 파이프라인 통합으로 코드 중복 제거 및 유지보수성 향상

### 1-2. OpenCASCADE WASM 빌드 및 Emscripten 버전 업그레이드

#### 이슈 사항

3D CAD 모델링에 필요한 OpenCASCADE 엔진을 브라우저에서 실행하기 위해 WebAssembly로 컴파일해야 했습니다. 기존 Emscripten 3.x 기반 빌드는 32비트 제한과 TypeScript 지원 부재 등의 한계가 있었습니다.

#### 문제점

- Emscripten 3.x의 32비트 메모리 제한으로 대형 CAD 모델 처리 시 메모리 부족 발생
- JavaScript↔C++ 간 바인딩에 TypeScript 타입 정의(d.ts)가 없어 프론트엔드 개발 시 타입 안전성 부재
- CMake 빌드 체계와 Emscripten 타겟 간 호환성 문제

#### 개선 사항

- 코딩 에이전트를 활용하여 **Emscripten 버전을 3 → 4 → 5로 단계적 업그레이드** 수행
- 64비트 메모리 지원 활성화로 대형 CAD 모델 처리 가능
- **TypeScript 타입 정의(d.ts) 자동 생성** 파이프라인 구축으로 프론트엔드 타입 안전성 확보
- CMake 빌드 체계를 Emscripten 타겟으로 재구성하고, React 컴포넌트의 WASM 로딩 패턴을 전환하여 **Next.js SSR 호환성** 확보 — dynamic import + Suspense로 hydration 경고 제거

#### 적용 결과

- 데스크탑 설치 없이 브라우저에서 3D CAD 모델링 가능 — **사용자 진입 장벽 완전 제거**
- 64비트 지원으로 대형 모델 처리 시 메모리 제한 해소
- d.ts 자동 생성으로 프론트엔드-WASM 간 타입 안전한 인터페이스 확보

### 1-3. PostHog 기반 제품 분석 체계 구축 및 데이터 기반 의사결정

#### 이슈 사항

초기 제품 단계에서 사용자가 시뮬레이션 워크플로우의 어느 단계에서 이탈하는지, 어떤 기능을 주로 사용하는지에 대한 데이터가 전혀 없어 제품 방향성 결정이 감에 의존하고 있었습니다.

#### 문제점

- 사용자 행동에 대한 정량적 데이터 부재로 제품 우선순위 결정이 주관적
- 신규 기능 출시 시 영향도를 측정할 수 없어 반복적 개선이 불가능
- 서버 사이드 이벤트(회원가입, 시뮬레이션 제출 등)가 추적되지 않아 전체 사용자 여정을 파악할 수 없음

#### 개선 사항

- **PostHog 분석 파이프라인을 처음부터 구축**: 클라이언트 초기화 → 사용자 식별 → 서버 사이드 이벤트 추적까지 단계적으로 확장
- 기존 Slack 웹훅 알림을 **PostHog 서버 사이드 이벤트**로 전환 — 회원가입, 로그인, 시뮬레이션 제출 등 핵심 이벤트를 통합 대시보드에서 분석
- 클라이언트 IP 추적으로 지역별·언어별 사용 패턴 분석 기능 추가

#### 적용 결과

- 사용자가 어떤 패턴으로 제품을 사용하고 **어느 단계에서 이탈하는지** 정량적으로 파악 가능 — 데이터 기반 제품 방향성 결정 체계 확립
- **어떤 국가·언어 사용자들이 주로 유입되는지** 파악하여 다국어 대응 및 타겟 시장 우선순위 도출
- 서버 사이드 이벤트 통합으로 회원가입부터 시뮬레이션 완료까지 전체 사용자 여정을 단일 대시보드에서 분석

### 1-4. AI 기반 CAE 시뮬레이션 워크플로우 챗봇 개발

#### 이슈 사항

CAE 시뮬레이션은 **유체 영역 생성 → 경계 조건 설정 → 메시 생성 → 시뮬레이션 실행 → 결과 확인**의 다단계 워크플로우로 구성됩니다. 전문 지식이 없는 사용자가 각 단계의 파라미터를 올바르게 설정하기 어려워 사용 진입 장벽이 높았습니다.

#### 문제점

- 시뮬레이션 파라미터 설정에 CAE 도메인 지식이 필요하여 비전문가의 독립 사용이 어려움
- 각 단계 간 상태 의존성이 복잡 — 유체 영역 변경 시 경계 조건과 메시가 모두 무효화되는데, 사용자가 이를 인지하지 못함

#### 개선 사항

- **Mastra AI 프레임워크 + AI SDK v5** 기반으로 CAE 시뮬레이션 워크플로우 챗봇을 구축
- 메시 실행, 시뮬레이션 실행, 도메인 초기화, 메시 품질 조회 등 **모듈러 도구(Tool) 아키텍처**로 설계 — 자연어로 시뮬레이션 파이프라인 제어 가능
- 클라이언트 사이드 도구 실행으로 실제 시뮬레이션 제어를 챗봇 인터페이스에서 직접 수행
- YAML 기반 프롬프트 관리 시스템과 의도 분류(intent classification) 구현
#### 적용 결과

- 비전문가도 자연어 대화만으로 시뮬레이션 파이프라인을 조작할 수 있어 **사용 진입 장벽 대폭 감소**
- 모듈러 도구 아키텍처로 새로운 시뮬레이션 기능 추가 시 도구만 구현하면 챗봇이 자동 활용

### 1-5. v1.0 릴리즈 — 프론트엔드 아키텍처 재설계 및 제품 완성도 향상

#### 이슈 사항

v1.0 출시를 앞두고 기존 프론트엔드 아키텍처에 근본적인 구조 문제가 있었습니다. 시뮬레이션 워크플로우의 각 단계(CAD 로딩, 메시 생성, 시뮬레이션 실행)가 WASM 모듈과 강하게 결합되어 있어, 단계 간 순서 의존성이 코드에 하드코딩된 상태였습니다. 또한 UI 일관성 부재, 인증 시스템 미비 등 제품 완성도를 높이기 위한 전방위적 개선이 필요했습니다.

#### 문제점

- 시뮬레이션 워크플로우의 각 단계가 WASM 모듈의 내부 상태에 직접 의존 — 단계 순서를 변경하거나 독립적으로 실행하는 것이 불가능한 구조
- JavaScript↔WASM 간 강한 결합으로 한쪽을 수정하면 다른 쪽도 반드시 수정해야 하는 높은 변경 비용
- UX를 고려하지 않은 UI 설계로 사용자가 워크플로우 진행 상태를 파악하기 어려움
- 컬러 팔레트, 컴포넌트 스타일이 통일되지 않아 제품 완성도 부족

#### 개선 사항

- 시뮬레이션 워크플로우를 **WASM 모듈과 분리된 상태 관리 계층**으로 재설계 — 각 단계가 독립적으로 실행·테스트 가능하도록 의존성 역전
- JavaScript↔WASM 간 **명확한 인터페이스 계층**을 도입하여 결합도 감소 — WASM 내부 변경이 UI 코드에 전파되지 않는 구조
- UX를 고려한 워크플로우 UI 재설계 — 단계별 진행 상태 시각화, 에러 피드백 개선
- **통합 컬러 팔레트 시스템**으로 전체 UI 색상 체계를 재설계하고, shadcn 컴포넌트를 재구성하여 디자인 시스템 표준화
- **Storybook** 통합으로 컴포넌트 단위 개발·문서화·인터랙션 테스트 환경 구축
- NextAuth 기반 인증 시스템 구축 — 로그인/로그아웃 리디렉션, 다국어(locale) URL 처리, CORS 설정
- Devcontainer, nginx 보안 헤더, GitHub Actions 워크플로우 등 DevOps 인프라 정비

#### 적용 결과

- WASM-JS 분리로 워크플로우 단계 간 **독립적 개발·테스트 가능** — 기능 추가 시 영향 범위 최소화
- UX 재설계로 사용자가 시뮬레이션 진행 상태를 직관적으로 파악 가능
- 디자인 시스템 표준화로 UI 일관성 확보 및 신규 화면 개발 속도 향상
- 안정적인 인증·인프라 체계 위에서 v1.0 릴리즈 완료

### 2-0. jQuery/PHP 레거시에서 React/Laravel 도입 주도

#### 이슈 사항

입사 당시 회사의 기술 스택은 jQuery와 프레임워크 없는 바닐라 PHP 5였습니다. 모던 PHP(7.x 이상)도 아닌 레거시 환경에서, 코드의 구조화가 부족하고 프론트엔드와 백엔드의 경계가 모호한 상태로 유지보수와 기능 추가에 큰 비용이 발생하고 있었습니다.

#### 문제점

- jQuery 기반 DOM 직접 조작으로 UI 상태 관리가 불가능 — 복잡한 화면에서 버그가 빈번히 발생
- 프레임워크 없는 바닐라 PHP 5로 MVC 패턴이 없어 비즈니스 로직, 데이터 접근, 뷰 렌더링이 한 파일에 혼재
- PHP 5의 보안 업데이트 종료로 보안 취약점 노출 위험

#### 개선 사항

- 지속적 건의를 통해 **React + Laravel** 도입을 관철하고, 전환 계획을 수립·실행
- 프론트엔드를 React로 전면 전환하여 컴포넌트 기반 UI 개발 체계 구축
- 백엔드를 Laravel로 마이그레이션하여 MVC 아키텍처, ORM(Eloquent), 라우팅 등 체계적 개발 환경 확보
- PHP 버전을 7.x → 8.x로 단계적 업그레이드

#### 적용 결과

- 컴포넌트 재사용과 상태 관리 도입으로 **프론트엔드 개발 속도 및 DX 대폭 향상**
- Laravel의 체계적 구조로 백엔드 유지보수성 개선
- 최신 PHP 버전 적용으로 보안 취약점 해소 및 성능 향상

### 2-1. 대량 급여 정산 화면의 프론트엔드 성능 최적화

#### 이슈 사항

초기 개발 시 소규모 사업장(수십 명)을 기준으로 설계되어, 서비스 성장 후 수천 명 규모의 사업장이 유입되면서 성능 문제가 발생했습니다. 급여 정산 기간에 수천 명의 근로자 데이터를 한 화면에서 조회·계산·수정해야 했으나, 대량 데이터를 고려하지 않은 설계로 테이블 렌더링 시 **UI가 수 초간 프리징**되어 사용자가 "화면이 멈췄다"는 CS를 반복 접수했습니다.

#### 문제점

- 전체 근로자 데이터를 한 번에 DOM에 렌더링하여 DOM 노드 수천 개 생성 → 브라우저 메인 스레드 블로킹
- 급여 계산 중 부모 컴포넌트 상태 변경이 전체 테이블 행의 불필요한 리렌더링을 유발
- 백엔드에서도 N+1 쿼리로 인해 API 응답 자체가 느렸음 — 근태 계산 로직이 복잡하고 동일 계산이 여러 API에서 중복 호출

#### 개선 사항

- **백엔드**: Redis 캐싱을 도입하여 반복 계산 결과 저장. N+1 문제를 해결하고 Eager Loading으로 전환. 20명 이하 소규모 사업장에서는 캐싱 오버헤드가 더 크므로 캐시 바이패스 로직 추가

### 2-4. CI/CD 파이프라인 및 개발 환경 최적화

#### 이슈 사항

#### 문제점

- GitHub-hosted runner의 네트워크/디스크 성능 제한으로, 매 실행마다 전체 의존성을 다운로드

#### 개선 사항

- **Self-hosted runner** 도입으로 빌드 환경을 직접 관리, 이후 **ARC(Actions Runner Controller)**로 업그레이드하여 Kubernetes 기반 러너 관리로 전환

- **Husky + lint-staged** 도입으로 커밋 전 파일 타입별 자동 검사 — TypeScript는 ESLint + Prettier, PHP는 Rector + PHPStan 정적 분석
- **Node.js 20** 업그레이드로 V8 엔진 개선에 따른 빌드 성능 향상

#### 적용 결과

- ARC 전환 + 3단계 병렬 실행으로 **CI 대기 시간 단축**, 중복 실행 방지로 비용 절감
- 커밋 전 자동 검사로 CI에서 발견되던 스타일·타입 오류가 **로컬에서 사전 차단**
- Node.js 20 업그레이드를 통해 빌드 최적화

### 3-1. Electron + React 실시간 대량 데이터 시각화

#### 이슈 사항

포스코 생산 라인에서 WebSocket으로 실시간 유입되는 철판 결함 데이터를 Electron 앱 내 React 대시보드에 즉시 렌더링해야 했습니다. 결함의 2차원 좌표, 크기, 등급 등을 시각화하고, 코일 단위로 품질 등급을 실시간 판정해야 했습니다.

#### 문제점

- WebSocket으로 유입되는 데이터 속도가 클라이언트 측 가공 속도를 초과 — 데이터가 쌓이면서 렌더링이 점점 지연
- 동일한 결함 데이터에 대해 여러 컴포넌트에서 중복 계산이 발생
- 모든 중간 가공 데이터를 디스크에 저장하고 있어, 운영 기간이 길어질수록 디스크 사용량이 지속 증가
- 코일 등급 판정 기준이 공정마다 상이하여 조건 분기가 매우 복잡

#### 개선 사항

- 수신 데이터에서 불필요한 항목을 걸러내는 **전처리 필터링 단계** 추가
- 가공 결과를 **메모이제이션**하여 동일 데이터의 재계산 방지, **배치 렌더링**으로 DOM 업데이트 최소화
- 중간 데이터를 디스크에 저장하지 않고 **메모리 내 파이프라인 방식**으로 전환 — 최종 결과만 영구 저장
- 등급 판정 로직을 **규칙 엔진 형태로 추상화** — 조건과 결과를 분리하고, 공정별 기준을 설정 파일로 관리

#### 적용 결과

- 데이터 유입 속도에 밀리지 않는 **실시간 처리 성능 확보**
- 메모리 파이프라인 전환으로 운영 중 디스크 사용량이 시간에 따라 증가하던 문제 해결
- 규칙 엔진 도입으로 신규 공정 대응 시 **코드 수정 없이 설정 파일만으로 대응** 가능
